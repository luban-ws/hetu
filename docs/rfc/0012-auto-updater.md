# 0012: Auto-Updater via GitHub Releases

**Status**: Draft  
**Date**: 2026-03-15  
**Author**: —  
**Supersedes**: RFC 0009 (updater portion only)

**Scope (one job)**: Integrate `tauri-plugin-updater` with GitHub Releases as the update endpoint, so Hetu can check for, download, and install updates in-app. Rewrite `UpdaterService` to use the Tauri updater API directly instead of the legacy IPC relay pattern.

## Summary

Add the Tauri updater plugin to the Rust backend and the `@tauri-apps/plugin-updater` + `@tauri-apps/plugin-process` JS packages to the frontend. Configure `tauri.conf.json` to validate updates against a signing key and pull the `latest.json` manifest from GitHub Releases. Rewrite the Angular `UpdaterService` to call the plugin API directly. Update the CI workflow to sign artifacts and upload the updater manifest.

## Motivation

- The Electron `electron-updater` was removed during the Tauri migration (RFC 0009). Users currently have no way to receive in-app updates.
- Tauri 2's updater plugin is purpose-built for this: it verifies signatures, supports GitHub Releases as a static endpoint, and requires zero server infrastructure.
- The existing `UpdaterService` and `UpdaterComponent` are already wired into the UI — only the transport layer (IPC-based relay) needs to be replaced with the native Tauri plugin API.

## Detailed Design

### Architecture

```
┌─────────────────────────────────────────────────────────────┐
│  Angular Frontend                                            │
│                                                              │
│  UpdaterService                                              │
│    → import { check } from '@tauri-apps/plugin-updater'      │
│    → import { relaunch } from '@tauri-apps/plugin-process'   │
│    → check() → download+install → relaunch()                │
└──────────────────────────┬──────────────────────────────────┘
                           │ Tauri plugin IPC (handled by plugin)
┌──────────────────────────▼──────────────────────────────────┐
│  Tauri Core (Rust)                                           │
│                                                              │
│  tauri-plugin-updater                                        │
│    → Fetches latest.json from GitHub Releases                │
│    → Verifies signature with embedded public key             │
│    → Downloads bundle, applies update                        │
│                                                              │
│  tauri-plugin-process                                        │
│    → Provides relaunch() capability                          │
└──────────────────────────┬──────────────────────────────────┘
                           │ HTTPS GET
┌──────────────────────────▼──────────────────────────────────┐
│  GitHub Releases                                             │
│                                                              │
│  latest.json (auto-generated by tauri-action)                │
│    { version, platforms: { "darwin-aarch64": { url, sig } }} │
│                                                              │
│  Signed bundles: .dmg, .msi, .AppImage, .deb + .sig files   │
└─────────────────────────────────────────────────────────────┘
```

### How GitHub Releases Updater Works

1. **Build time**: `tauri build` with `TAURI_SIGNING_PRIVATE_KEY` env var produces signed bundles (`.sig` files alongside each installer).
2. **CI release**: `tauri-apps/tauri-action` uploads bundles + signatures to the GitHub Release and auto-generates `latest.json` with platform-specific download URLs and embedded signatures.
3. **Runtime**: The updater plugin fetches `https://github.com/luban-ws/hetu/releases/latest/download/latest.json`, parses the version, and compares against the running app version. If newer, it downloads the platform-specific bundle, verifies the signature against the embedded public key, installs, and relaunches.

### Signing Key Management

| Key | Location | Usage |
|-----|----------|-------|
| **Private key** | GitHub Actions secret `TAURI_SIGNING_PRIVATE_KEY` | Signs bundles at build time |
| **Private key password** (optional) | GitHub Actions secret `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` | Unlocks the private key |
| **Public key** | `tauri.conf.json` → `plugins.updater.pubkey` | Embedded in the app binary for verification |

The private key is generated once with `pnpm tauri signer generate` and stored as a GitHub secret. It must **never** be committed to the repo. The public key is safe to commit.

### UpdaterService Rewrite

The current `UpdaterService` uses the IPC relay pattern (`adapter.send('Updater-Check')` → backend checks → backend emits events). The Tauri updater plugin exposes a direct JS API — no custom IPC needed.

**Current flow** (legacy, IPC-based):
```
UpdaterService.checkUpdate()
  → adapter.send('Updater-Check')
  → Rust/Electron checks for update
  → emits 'Updater' event with { msg: 'update-available', version }
  → emits 'Updater' event with { msg: 'downloading-update', percentage }
  → emits 'Updater' event with { msg: 'download-complete' }
  → adapter.send('Updater', 'commence-install-update')
```

**New flow** (direct plugin API):
```
UpdaterService.checkUpdate()
  → import { check } from '@tauri-apps/plugin-updater'
  → const update = await check()
  → update.downloadAndInstall(onProgress)
  → import { relaunch } from '@tauri-apps/plugin-process'
  → await relaunch()
```

The rewritten service keeps the same public API (`checkUpdate()`, `installUpdate()`, `isUpdateAvailable`, `updateVersion`, event emitters) so `UpdaterComponent` requires no changes.

### CI Workflow Changes

The existing `.github/workflows/release.yml` needs:

1. `TAURI_SIGNING_PRIVATE_KEY` and `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` environment variables from GitHub secrets.
2. The `tauri-action` already sets `uploadUpdaterJson: true` by default, so `latest.json` is auto-generated and uploaded.
3. Change `releaseDraft: true` to `releaseDraft: false` (or manually publish drafts) so that `releases/latest/download/latest.json` resolves correctly. Alternatively, use a tag-based endpoint.

```yaml
- name: Build and release
  uses: tauri-apps/tauri-action@v0
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
    TAURI_SIGNING_PRIVATE_KEY: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY }}
    TAURI_SIGNING_PRIVATE_KEY_PASSWORD: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY_PASSWORD }}
  with:
    # ... existing config ...
    updaterJsonPreferNsis: true
```

### `latest.json` Format (auto-generated)

The `tauri-action` generates this file and attaches it to the GitHub Release:

```json

  "version": "0.6.0",
  "notes": "Release notes from GitHub",
  "pub_date": "2026-03-15T12:00:00Z",
  "platforms": {

      "signature": "<base64 sig content>",
      "url": "https://github.com/luban-ws/hetu/releases/download/v0.6.0/Hetu_0.6.0_aarch64.app.tar.gz"
```

## Alternatives Considered

| Option | Pros | Cons | Decision |
|--------|------|------|----------|
| CrabNebula Cloud | Managed service, analytics | External dependency, cost | Rejected — GitHub is already the release host |
| Self-hosted update server | Full control | Maintenance overhead, needs hosting | Rejected — GitHub Releases is zero-cost |
| Static JSON on S3/CDN | Fast, redundant | Extra infra to manage, manual upload | Rejected — tauri-action auto-generates JSON |
| GitHub Releases + `latest.json` | Zero infra, auto-generated by CI | Tied to GitHub, public repo needed for public releases | **Accepted** |

## Migration Strategy

1. The existing `UpdaterService` public API (`checkUpdate`, `installUpdate`, event emitters) remains unchanged. `UpdaterComponent` needs no modifications.
2. The IPC channels `Updater-Check`, `Updater-Checking`, and the `Updater` event channel become unused and can be removed from `ipc-events.ts` after this RFC is complete.
3. Users on versions before this update will not auto-update (they were already on Electron). The first Tauri release with the updater is the baseline — subsequent releases will auto-update.

## Testing Strategy

| Test | How |
|------|-----|
| Signing key generation | Run `pnpm tauri signer generate -- -w ~/.tauri/hetu.key` locally; verify key pair created |
| Local signed build | Set `TAURI_SIGNING_PRIVATE_KEY` env, run `pnpm tauri build`; verify `.sig` files alongside bundles |
| CI build + latest.json | Push a `v*` tag; verify GitHub Release contains `latest.json` with correct platform entries |
| Update check (no update) | Run app at current version; `check()` returns `null` |
| Update check (update available) | Publish a newer version release; run older app; `check()` returns update object with correct version |
| Download + install | Call `downloadAndInstall()`; verify progress callbacks fire; app restarts to new version |
| Signature verification failure | Tamper with `latest.json` signature; verify update is rejected |
| Offline / network error | Disconnect network; verify `check()` rejects gracefully, UI shows error toast |

## Implementation Plan (Detailed Steps)

### 1. Generate Signing Keys

1.1. Run `pnpm tauri signer generate -- -w ~/.tauri/hetu.key` to generate private + public key pair.  
1.2. Copy the **public key** content (displayed in terminal output).  
1.3. Store the **private key** file content (`~/.tauri/hetu.key`) as GitHub repository secret `TAURI_SIGNING_PRIVATE_KEY`.  
1.4. If a password was set, store it as GitHub secret `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`.  
1.5. **IMPORTANT**: Back up the private key securely. If lost, existing users cannot receive updates.

### 2. Add Rust Dependencies

2.1. In `apps/desktop/src-tauri/Cargo.toml`, add:
```toml
tauri-plugin-updater = "2"
tauri-plugin-process = "2"
```

2.2. In `apps/desktop/src-tauri/src/lib.rs`, register plugins:
```rust
.plugin(tauri_plugin_updater::init())
.plugin(tauri_plugin_process::init())
```

### 3. Configure tauri.conf.json

3.1. Add `"createUpdaterArtifacts": true` to `bundle` section.  
3.2. Add `plugins.updater` section with the **public key** and GitHub endpoint:
```json
{
  "plugins": {
    "updater": {
      "pubkey": "<PUBLIC_KEY_FROM_STEP_1.2>",
      "endpoints": [
        "https://github.com/luban-ws/hetu/releases/latest/download/latest.json"
      ],
      "windows": {
        "installMode": "passive"
      }
    }
  }
}
```

### 4. Update Capabilities

4.1. In `apps/desktop/src-tauri/capabilities/default.json`, add permissions:
```json
"updater:default",
"process:allow-restart"
```

### 5. Install Frontend Packages

5.1. Run `pnpm --filter @luban-ws/desktop add @tauri-apps/plugin-updater @tauri-apps/plugin-process`

### 6. Rewrite UpdaterService

6.1. Replace `apps/desktop/src/renderer/app/infrastructure/updater.service.ts` with the new implementation that:
- Imports `check` from `@tauri-apps/plugin-updater`
- Imports `relaunch` from `@tauri-apps/plugin-process`
- `checkUpdate()`: calls `check()`, sets `isUpdateAvailable` and `updateVersion`
- `installUpdate()`: calls `update.downloadAndInstall()` with progress callback, then `relaunch()`
- Preserves existing public API: `isUpdateAvailable`, `updateVersion`, `updateChecking`, `updateAvailableChange` emitters
- Removes all `adapter.on('Updater', ...)` and `adapter.send('Updater-Check')` calls

6.2. Update `updater.service.spec.ts` to mock `@tauri-apps/plugin-updater` and `@tauri-apps/plugin-process`.

### 7. Clean Up Legacy IPC Channels

7.1. Remove `UPDATER` section from `apps/desktop/src/renderer/app/infrastructure/ipc-events.ts`.  
7.2. Remove `UPDATER` from the `IPC_EVENTS` export object.  
7.3. Remove any `IPC_EVENTS.UPDATER` imports in other files (search and verify).

### 8. Update CI Workflow

8.1. In `.github/workflows/release.yml`, add environment variables to the build step:
```yaml
env:
  GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
  TAURI_SIGNING_PRIVATE_KEY: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY }}
  TAURI_SIGNING_PRIVATE_KEY_PASSWORD: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY_PASSWORD }}
```

8.2. Add `updaterJsonPreferNsis: true` to the `with` block.  
8.3. Consider changing `releaseDraft: true` to `releaseDraft: false` so `releases/latest/download/latest.json` resolves automatically. Alternatively, keep draft mode and manually publish — but `latest` won't resolve until published.

9.2. `pnpm desktop:build` — no errors.  
9.3. Local signed build: set `TAURI_SIGNING_PRIVATE_KEY` env, run `pnpm tauri build`; verify `.sig` files generated.

## Open Questions

1. **Release strategy**: Should we keep `releaseDraft: true` and manually publish releases, or switch to auto-publish (`releaseDraft: false`)? Manual publish gives a review step; auto-publish enables `releases/latest` immediately.
2. **Update prompt UX**: Should the user be prompted to install immediately, or should it auto-install on next launch? Current UI shows a toast with a click-to-install pattern.
3. **Key backup**: Where should the signing private key be backed up beyond GitHub Secrets? (e.g., team password manager, encrypted USB).
4. **macOS code signing**: Separate from updater signing — do we need an Apple Developer certificate for notarization? This is a prerequisite for macOS distribution but out of scope for this RFC.
